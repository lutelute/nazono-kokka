# ベクトル空間ガイド — RAG を「桃太郎で」理解する

> このドキュメントは、「謎の国家」の RAG パイプラインを支える
> **ベクトル空間** の考え方を、コードを書かずに掴むためのものです。
> Streamlit の **ベクトル可視化** ページや
> `python -m rag_system.vector_analysis` の CLI を触る前後に読むと、
> 何が起きているのかが立体的に見えてきます。

---

## 0. そもそも RAG とは何か

**RAG = Retrieval-Augmented Generation**（検索で補強した生成）。
LLM 単体の弱点を「書庫からの検索」で補う仕組みです。

| LLM 単体の弱点 | RAG が解決する仕組み |
|---|---|
| 学習時にない最新知識を知らない | **書庫から検索** して都度プロンプトに混ぜる |
| 自信満々に嘘をつく（ハルシネーション） | **実在する文書だけ** を根拠に答えさせる |
| ドメイン固有知識（謎の国家の刑法など）が薄い | 専門書庫を **ベクトル DB** に詰めて引かせる |

### 4 つのステップ

```
①埋め込み      ②検索           ③プロンプト構築    ④生成
──────────    ──────────       ─────────────     ──────────
質問テキスト → 質問ベクトル → 上位 K 件文書 → 大きな文章 → LLM 回答
(768 次元)    (1354 件と比較)  (条文+判例)     (テンプレート)
```

1. **①埋め込み** : 文章を 768 個の数字（ベクトル）に変える。
   このプロジェクトは `sentence-transformers/paraphrase-multilingual-mpnet-base-v2` を使用。
   取り込み時と質問時で **同じモデル** を使うのがミソ（同じ空間に乗せるため）。
2. **②検索** : 質問ベクトルと書庫の 1,354 チャンクすべてとの
   **コサイン類似度** を計算し、上位 K 件を取り出す。これが RAG の心臓部。
3. **③プロンプト構築** : 取り出したチャンクを `{context}` に、
   質問を `{question}` に差し込んだ大きなプロンプトを作る（LangChain "stuff" chain）。
4. **④生成** : そのプロンプトを Ollama LLM に渡し、回答テキストを得る。
   プロンプトに条文番号・判例 ID が入っているので、LLM は引用付きで答えられる。

このプロジェクトの実装は `rag_system/judge.py` に集約されており、
`JUDICIAL_PROMPT_TEMPLATE` を読むと ③ で何が組み立てられているかが分かります。
**Streamlit のベクトル可視化ページ・セクション 0** ではこの 4 工程を
インタラクティブに 1 ステップずつ確認できます。

## 1. なぜベクトルなのか

LLM が「窃盗罪の量刑」という質問に答えるとき、
裏では次の流れが走っています。

1. **質問を 768 個の数字に変換する**（埋め込み）
2. その数字列に **近い** 法令や判例を書庫から探す（ベクトル検索）
3. 引いてきた条文を LLM に渡して回答を生成させる（生成）

つまり RAG の精度は、**「近い／遠い」の幾何学** がすべてです。
ここでいう「近い」を直観的に掴むには、童話で例えるのが手っ取り早い。

## 2. 桃太郎 vs 浦島太郎 — 似ているとは何か

`scripts/vector_space_demo.py` を実行すると、
次のような類似度マトリックスが出ます。

|        | 桃太郎 | 浦島太郎 | 三匹の子豚 | シンデレラ |
|--------|--------|----------|------------|-----------|
| 桃太郎 | 1.000  | **0.700** | 0.580      | 0.469     |
| 浦島太郎 | **0.700** | 1.000 | 0.555      | 0.569     |
| 三匹の子豚 | 0.580 | 0.555 | 1.000      | 0.314     |
| シンデレラ | 0.469 | 0.569 | 0.314      | 1.000     |

数字は **コサイン類似度**（−1 ～ +1、大きいほど似ている）。
直感的な解釈：

- **桃太郎と浦島太郎 = 0.700**
  どちらも「若い男性が異界へ旅立ち、何かを得て帰ってくる」物語。
  主要モチーフ（旅・主人公・異界）が共通するため強く類似。

- **桃太郎と三匹の子豚 = 0.580**
  「童話」という大きなジャンルは共通だが、物語類型が違う。
  共通要素はあるが個別の構造（家を建てる vs 鬼退治）はまったく別。

- **桃太郎とシンデレラ = 0.469**
  さらに離れる。「主人公の成長」しか共通要素がない。

- **三匹の子豚とシンデレラ = 0.314**
  同じ童話なのに最も離れている。
  物語類型が「家とオオカミ」vs「魔法と王子」で全くの別物。

> **ポイント**：人間が直感的に持つ「似てる／似てない」の感覚と、
> 埋め込みベクトルが算出するコサイン類似度はおおむね一致します。
> 完璧に一致するわけではないが、RAG の挙動を理解するには
> この一致度合いを掴むのが最初の一歩です。

## 3. 謎の国家の書庫はどう並んでいるか

童話と同じことを「謎の国家」の 1,354 チャンクでやると：

| | 憲法 | 刑法 | 民法 | 判例(刑事) | 判例(民事) | 判例(憲法) |
|---|------|------|------|-----------|-----------|------------|
| 憲法 | 1.00 | 0.55 | 0.60 | 0.45 | 0.55 | 0.75 |
| 刑法 | 0.55 | 1.00 | 0.70 | 0.80 | 0.50 | 0.45 |
| 民法 | 0.60 | 0.70 | 1.00 | 0.55 | 0.85 | 0.50 |
| 判例(刑事) | 0.45 | **0.80** | 0.55 | 1.00 | 0.50 | 0.40 |
| 判例(民事) | 0.55 | 0.50 | **0.85** | 0.50 | 1.00 | 0.45 |
| 判例(憲法) | **0.75** | 0.45 | 0.50 | 0.40 | 0.45 | 1.00 |

（実際の数値はベクトル可視化ページの **セクション 4** で見られます）

期待される関係がきれいに出ています：

- **刑法 ⇔ 刑事判例 = 0.80**（刑法の条文と、その適用結果である判例）
- **民法 ⇔ 民事判例 = 0.85**（民法の条文と、その適用結果である判例）
- **憲法 ⇔ 憲法判例 = 0.75**（憲法の条文と、その適用結果である判例）

そして童話との関係は：

- **桃太郎 → 最も近い法律チャンク = 0.39 程度**
- **法律クエリ「窃盗罪の量刑」→ 最も近い刑法チャンク = 0.70 程度**

この **0.39 vs 0.70** の差が、RAG が「童話質問には童話を、法律質問には法律を引いてくる」根拠です。

## 4. 次元削減 — 768 次元を「目」で見る

埋め込みは 768 次元です。
私たちが見られるのは 2D / 3D だけなので、
**次元削減** が必要になります。

### PCA（主成分分析）
- **線形**で**高速**。全体構造を保つ。
- 軸の意味：データの分散が最大になる方向。
- このページの「書庫マップ」で **PCA** を選んだ場合、
  「法令ぜんたい vs 判例ぜんたい」のような **大きな差** が
  軸として現れます。

### t-SNE（t-distributed Stochastic Neighbor Embedding）
- **非線形**で**低速**（しかし美しい）。
- 軸の意味：近い点はできるだけ近く、遠い点はできるだけ遠く。
- 「書庫マップ」で **t-SNE** を選ぶと、
  **小さなクラスタ**（窃盗の判例だけが集まる、表現の自由の判例だけが集まる、など）が
  鮮明に見えます。

両方試して、PCA は森を、t-SNE は木を見るためのレンズだと思ってください。

## 5. クラスタリング — メタデータを使わない群分け

ベクトル可視化ページの **セクション 3**（クラスタ自動発見）では、
**人手のラベル（憲法／刑法／判例…）を一切使わず**、
**K-means** だけで書庫をグループ分けします。

期待される結果は：

- 人手ラベルとクラスタが **ほぼ一致** すれば、
  → RAG の意味理解は健全。「刑法」と書いたら刑法の話がまとまってくる。
- 一つの人手ラベルが **複数クラスタに分散** すれば、
  → そのラベル内に **隠れたサブトピック** がある。
  例：「刑事判例」のなかに「窃盗系」「暴力系」「経済犯罪系」が同居していて、
  ベクトル空間ではそれぞれ別の場所に座っているかもしれない。

この **クロス集計** が、書庫の意味的な健康診断になります。

## 6. ベクトル算術 — 意味の足し算引き算

有名な例：

```
ベクトル(王) − ベクトル(男) + ベクトル(女) ≒ ベクトル(女王)
```

これと同じことを文章でやるのが **セクション 8（ベクトル算術ラボ）** です。

- **「刑法」 + 「量刑」**
  → 結果ベクトルを書庫で検索すると、**刑事判例**（特に量刑が争点のもの）が上位に。
- **「桃太郎」 + 「海」**
  → **浦島太郎**（海＋主人公の冒険）に近づく、はず。

このように **意味の足し算** がうまく機能するということは、
ベクトル空間に **言語の意味構造** が幾何学的に埋め込まれている、ということです。

## 7. RAG の境界はどこにあるか — クロスドメイン・プローブ

ベクトル可視化ページの **セクション 7-d** では、
**童話をクエリにして謎の国家の書庫を検索** できます。

期待される挙動：

- 桃太郎を投げる → 最大類似度 0.30〜0.40 程度
- 浦島太郎を投げる → 同程度
- 法律クエリを投げる → 最大類似度 0.60〜0.80

この **0.40 と 0.70 の間** が、RAG の **意味境界** です。
0.5 を閾値にすれば、「童話質問に対しては引かない」というガードレールが
ベクトル幾何だけで作れます。

実プロダクトでこの閾値を設定するときは、
**ベクトル可視化ページの類似度分布ヒストグラム** が役立ちます。

## 8. まとめ — ここから RAG をどう調整するか

1. **チャンクサイズ**：大きすぎる → ベクトルが「平均化」されて意味がぼやける。
   小さすぎる → 文脈を失う。
   → ベクトル可視化ページでクラスタの分かれ方を見ると最適サイズの当たりが付く。

2. **埋め込みモデル**：今は `paraphrase-multilingual-mpnet-base-v2`（768d）。
   日本語の法律ドキュメントには十分。
   もし「童話と法律の境界が甘い」と感じたら、専門ドメインのモデルに変える検討。

3. **取得件数 K**：5〜10 がふつう。
   類似度分布ヒストグラム（クエリ近傍探索の下部）で、
   類似度 0.5 を超える文書がいくつあるか確認して決める。

4. **メタデータフィルタ**：ベクトル類似度だけでなく
   「刑事判例だけから引く」のような **メタフィルタ** を併用すれば、
   ヒット率がさらに上がる。RAG ページの判例検索ツールはこの併用パターン。
